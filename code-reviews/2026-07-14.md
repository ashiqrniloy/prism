# Prism code review — 2026-07-14

## Scope

Reviewed `@arnilo/prism` 0.0.3 core, first-party provider/compaction/coding-agent workspaces, tests, docs, packaging, and release workflow. Focus: functionality, reuse/maintainability, performance, security, missing capabilities, and `prism-bug-report.md`.

## Verification performed

- `npm run sdk:ready` — **pass** in 24.5s (typecheck, build, network-free tests, workspace tests, install/package checks, pack dry-runs).
- `npm audit --json` — **0 known vulnerabilities** across 29 installed dependencies.
- `npm outdated --json` — patch update available for `@types/node`; major versions available for `diff` and TypeScript. No release-blocking update identified.
- Reviewed package/version/export maps and release workflow. All 13 package manifests are at `0.0.3`; first-party peer/dependency pins are consistent.
- Current tests do **not** cover reported redactor + revision-loop failure or request-level repair-message uniqueness.

## Executive assessment

Prism has strong boundaries, strict TypeScript, broad network-free tests, explicit credential ownership, package export checks, and good release packaging. Release 0.0.4 should not ship until revision-loop transcript ownership, graph redaction, malformed-message diagnostics, and multi-round tool transcript ordering are fixed.

## Bug report evaluation

**Verdict: valid and reproducible from code.** Root-cause analysis is accurate.

1. `src/agent-loops.ts:153-158` inserts each repair `Message` object into `ctx.history`, then reuses the same objects as `nextInput`.
2. `src/input.ts:91-101,225-231` flattens history and input into one request, so the repair appears twice with shared identity.
3. `src/redaction.ts:43-53` uses one traversal-wide `WeakSet`; any repeated reference, not only an ancestor cycle, becomes `"[Circular]"`.
4. `src/providers/openai-compatible.ts:135-151` assumes every message is an object with array `content`, causing the reported native `TypeError`.

Fix A from the report prevents shared identity but leaves duplicate revision instructions. Better root fix: mirror first-turn ordering—persist repair input, assemble it exactly once outside live history, then add it to live history only after that request has been generated. This resolves A and B together. Fix C and D remain necessary defense-in-depth.

## Findings

### P0 — Revision request is duplicated and corrupted by redaction

**Locations:** `src/agent-loops.ts:153-158`, `src/input.ts:91-101`, `src/redaction.ts:43-53`, `src/providers/openai-compatible.ts:135-151`

**Impact:** Any generate→validate→revise run with an active secret redactor crashes on first revision. Without a redactor, model still receives repair instruction twice.

**Recommendation:** Change revision history timing, add end-to-end request-shape regression tests, make graph redaction path-aware, and validate provider message shape with index/path diagnostics.

### P1 — Multi-round tool transcript becomes chronologically invalid

**Locations:** `src/agent-loops.ts:43-67`; explicit current assumption in `src/__tests__/agent-loops.test.ts:75-83`

`singleShotLoop` appends tool results to store but never pushes them into live history. It instead accumulates all results in a separate array passed to every assembly. With `maxToolRounds > 1`, turn 3 is ordered approximately as:

```text
user, assistant(tool call 1), assistant(tool call 2), tool result 1, tool result 2
```

Tool result 1 must follow assistant call 1 before assistant call 2. Some providers reject malformed ordering; others infer incorrect context.

**Recommendation:** Add each persisted tool-result message to live history immediately and stop replaying an accumulating result array. Test three turns with two tool rounds and assert exact message order.

### P1 — Redactor can leak secrets in object/Map keys

**Location:** `src/redaction.ts:51-53`

Values are redacted, but object keys and `Map` keys are copied unchanged. `JSON.stringify(redactSecrets({ [secret]: true }, [secret]))` still contains the secret.

**Recommendation:** Redact string keys while handling collisions deterministically; test objects and Maps. Document that symbols/non-enumerable properties remain outside JSON-shaped redaction.

### P1 — Provider streams and error bodies are unbounded

**Locations:** `src/providers/openai-compatible.ts:194-215`; provider package `sse.ts` files; provider `safeText()` helpers

SSE readers retain an incomplete frame without a size ceiling, and HTTP error helpers consume full bodies. A malicious or broken endpoint can grow process memory until termination. Root OpenAI-compatible parser also handles individual `data:` lines rather than complete SSE events, so legal multiline `data:` events fail JSON parsing.

**Recommendation:** Reuse one bounded SSE frame reader across OpenAI-style packages, cap frame/error-body bytes, support multiline `data:` fields, cancel/release readers, and emit attributable protocol errors.

### P1 — Event-ledger writes have no backpressure

**Locations:** `src/agents.ts:361-374`

Every streamed event immediately starts an append and is retained in `ledgerPromises` until drain. Long outputs can create thousands of concurrent database writes and promises, defeating stream backpressure.

**Recommendation:** Serialize ledger appends through one promise chain or drain in a small bounded batch. Preserve event order and propagate failures at run completion. Benchmark with a high-delta mock stream.

### P1 — OpenAI device-code OAuth flow does not poll

**Location:** `packages/provider-openai/src/oauth.ts:79-102`

After obtaining a device code and notifying the host, implementation calls token endpoint once. Device authorization normally requires polling while authorization is pending, respecting interval/expiry and slow-down responses. Current flow will commonly fail before user can approve. Authorization/device codes are also not included in token-error redaction secrets.

**Recommendation:** Implement bounded abortable polling from returned interval/expiry fields; classify pending/slow-down responses; redact authorization code, device code, access token, and refresh token from failures.

### P2 — Provider adapters duplicate protocol utilities

**Locations:** all `packages/provider-*/src` provider/SSE modules and `src/providers/openai-compatible.ts`

Static scan found 9 `safeText`, 8 `parseArgs`, 7 `toTool`, 6 `toUsage`, plus six near-identical SSE readers and repeated OpenAI message serializers. Fixes and safety limits must currently be copied across packages, inviting drift.

**Recommendation:** Extract only proven shared primitives: bounded SSE frames, bounded response text, JSON-object argument parsing, and OpenAI-style message/tool serialization. Keep provider-specific request fields and event mapping local.

### P2 — JSONL append is O(file size) and corruption is silent on runtime paths

**Locations:** `src/node/session-store-jsonl.ts:34-54,82-86`

Each append rereads and reparses the full file for duplicate/parent checks. Runtime `list`, `get`, and append consume only valid entries and discard parse errors; append can continue after a corrupt/truncated line. In-memory idempotency state also disappears on restart.

**Recommendation:** Keep JSONL explicitly development/single-process, but fail append when parse errors exist and document restart idempotency limits. Production users should use indexed database adapters; do not turn JSONL into a database.

### P2 — Coding-agent image read ignores its public option and has no size limit

**Locations:** `packages/coding-agent/src/read.ts:176,239-249`; `docs/coding-agent-tools.md:89`

`autoResizeImages` is accepted but is a no-op. Entire image is read and base64-expanded into context, allowing accidental memory/token blowups.

**Recommendation:** For next compatible release, enforce a configurable byte ceiling and reject oversized images clearly. Either implement resize in a separate optional adapter or remove/deprecate the no-op option rather than implying behavior.

### P2 — Optional config-file detection depends on error text

**Locations:** `src/node/config.ts:39-54`, `src/node/settings.ts:24`

Missing optional files are detected with `error.message.includes("ENOENT")` after wrapping errors. This is brittle across custom errors/platforms and duplicated.

**Recommendation:** Preserve/check typed Node `code === "ENOENT"` before wrapping, using one small helper.

### P2 — Release workflow can publish a tag/version mismatch and partially publish a set

**Location:** `.github/workflows/release.yml`

Any `v*` tag triggers publishing; workflow does not assert tag equals root/all workspace versions. Publishing 13 packages is not transactional, so a later failure leaves a partial release. `id-token: write` is enabled but provenance is not requested.

**Recommendation:** Add version/tag/pin consistency gate before publish, registry collision preflight, explicit package order, `--provenance`, and documented resume procedure for partial publication.

## Reuse and maintainability assessment

### Strengths

- Core/platform split is clear: filesystem code stays under `src/node`; optional coding tools are separate.
- No hidden provider/credential global state.
- Strict TypeScript and frozen public export tests reduce accidental API drift.
- Conformance helpers, package smoke tests, docs checks, and network-free defaults are reusable and effective.
- Configuration merge rejects prototype-pollution keys.

### Main debt

- Provider transport/serialization duplication is largest repeated implementation surface.
- `src/contracts.ts` (1,323 lines), `src/agents.ts` (676), and `src/node/agent-definitions.ts` (520) are navigable but approaching change-conflict hotspots. Split only by cohesive domain when next changes touch them; no speculative class hierarchy.
- Many historical phase-boundary source-text tests protect architecture but are brittle under harmless refactors. Prefer behavioral/type/export tests where possible.

## Performance assessment

- SDK readiness is healthy locally at 24.5s, below documented CI backstop.
- Streaming provider consumption and subscriber queue limits are good foundations.
- Main scale risks: unbounded SSE/error input, unbounded ledger append concurrency, O(n²)-over-session JSONL appends, full recursive redaction clones, and raw image base64 expansion.
- Do not optimize registry/map lookups or prompt builders; no evidence they matter.

## Security assessment

### Strengths

- `npm audit`: zero known vulnerabilities.
- Provider-owned auth headers override caller headers.
- Credential resolution is explicit and late.
- Config keys reject `__proto__`, `prototype`, and `constructor`.
- Contribution discovery uses realpath containment and explicit trust/permission seams.
- Coding tools are opt-in and excluded from `prism-all`.

### Risks requiring action

- Redactor key leakage and destructive diamond handling.
- Unbounded remote response consumption.
- OAuth temporary-code error leakage and incomplete device flow.
- Coding tools intentionally permit shell execution and absolute filesystem access. This is documented, but hosts need a first-class approval/sandbox integration package before broad production use.

## Missing features/functionality

These are capability gaps, not all core-package requirements. Prefer optional packages and existing extension seams.

| Rank | Capability | Importance / relevance | Suggested home |
| ---: | --- | --- | --- |
| 1 | Runtime JSON Schema validation for tool arguments | High security/correctness. `parameters` is metadata today; malformed model calls reach tools unless host supplies a validator. | Core using host-injected validator, plus optional standards adapter; avoid mandatory schema dependency. |
| 2 | Native provider structured-output / JSON-schema controls | High. Current artifact loop repairs text after generation and costs extra turns; providers can constrain output directly. | Provider request options/capabilities + provider packages. |
| 3 | MCP client bridge | High ecosystem relevance; unlocks standard remote/local tool servers without bespoke adapters. | Separate opt-in package. |
| 4 | Shared resilient provider transport | High. Bounded SSE, timeout, retry hints, rate-limit metadata, protocol diagnostics, and tracing should behave consistently. | Small reusable provider utility package or public core primitives. |
| 5 | Production database adapters | High adoption relevance despite host-owned persistence boundary. Reference PostgreSQL/SQLite adapters would reduce correctness risk around atomic append, idempotency, tenancy, and queries. | Separate packages. |
| 6 | Human approval and sandbox policy for coding tools | High security relevance for shell/write/edit. Existing permission policy decides by tool name, not command/path/risk. | Separate host-security/coding-agent integration package. |
| 7 | Parallel tool-call execution with ordering controls | Medium-high. Models emit parallel calls; current loop dispatches sequentially. Needs opt-in concurrency and deterministic transcript ordering. | Core loop option. |
| 8 | Provider/tool observability hooks | Medium-high. OpenTelemetry spans/metrics, request IDs, latency, retries, token/cost/energy, and redacted diagnostics. | Optional integration package over existing events/middleware. |
| 9 | Workflow/graph orchestration | Medium. Useful for larger apps, but Prism's bounded loop strategy is intentionally smaller and generic. | External extension package, not core. |
| 10 | Audio/file/document multimodality | Medium. Content contracts and providers mostly cover text/image; audio and document uploads are absent. | Contracts only when two providers need them, then provider packages. |
| 11 | Persistent encrypted credential/keychain adapters | Medium-low for framework core; useful for CLI apps but conflicts with host-owned credential design. | Separate platform packages. |
| 12 | Interactive TUI | Low for SDK relevance; CLI/RPC already provide host integration seams. | External app/package. |

## Release recommendation

**Hold 0.0.4** until P0 and P1 findings have regression tests and release gates pass. P2 items may ship as documented follow-ups except tag/version consistency, which should be part of release readiness. No dependency upgrade is required for the bug-fix release.
